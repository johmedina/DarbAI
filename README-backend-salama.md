# DarbAI Backend

FastAPI backend for **DarbAI**, an assistant that helps users prepare for the
driving theory test in Qatar. It answers questions about road rules, traffic
signs, penalties and safety based on the official *Driving Guide in the State
of Qatar*, in **English or Arabic**.

Five things make this backend more than a chat wrapper:

1. **Local generation with uncertainty quantification (UQ).** Answers are
   generated by a locally hosted `QCRI/Fanar-1-9B-Instruct` model (4-bit NF4
   quantized for fast inference). For every generated token the API reports
   entropy / reliability signals (including GLU and LogTokU aggregates), so
   the frontend can show how confident the model was.
2. **Self-hosted multimodal RAG.** The driving guide is ingested into a local
   index: text chunks *and* a catalog of every named traffic sign in the book
   (cropped image + name exactly as printed). Retrieval, embedding and search
   all run on this machine — no external search service.
3. **Sign images in both directions.**
   - User sends a **photo of a traffic sign** → the system identifies it and
     answers with the sign's exact name from the book.
   - User asks to **see** a sign ("show me the No Entry sign") → the response
     carries the actual sign image cropped from the book, served by this API.

   These work inside the generic chat mode, and each is also exposed as a
   **dedicated, LLM-free endpoint** for sign-only chat modes — `identify-sign`
   (photo → name + crop) and `find-sign` (description → crop). See §4.
4. **User accounts and persistent chat history.** JWT auth (signup/login),
   with every chat, message, regenerated version and uploaded image stored
   per user — PostgreSQL for data, Azure Blob Storage for images. All chat
   endpoints enforce ownership.
5. **Streaming generation.** Token-by-token SSE streaming with the full UQ
   metrics delivered in a final `done` event; previous answers can be
   regenerated as new versions.

## 1. Architecture

```
                       ┌────────────────────────────────────────────┐
                       │                FastAPI app                 │
                       │                                            │
 user question ──────► │  routes.py ─► generation_service.py        │
 (+ JWT, opt. image)   │   │             │        │                 │
                       │   │             │        ├─ retrieval_service.py
                       │   │             │        │    ├─ BGE-M3 (text embeddings, local GPU)
                       │   │             │        │    ├─ SigLIP (image embeddings, local GPU)
                       │   │             │        │    └─ Postgres `rag` schema (pgvector)
                       │   │             │        │
                       │   │             │        └─ Azure OpenAI GPT-4.1-mini
                       │   │             │           (vision only: photo re-rank / description)
                       │   │             │
                       │   │             └─ Fanar-1-9B (local GPU) + uq_service.py
                       │   │
                       │   ├─ auth_service.py        (JWT + bcrypt, `users` table)
                       │   ├─ Postgres `public`      (chats, messages, message_versions)
                       │   └─ azure_blob_service.py  (uploaded chat images)
                       └────────────────────────────────────────────┘
```

| Component | Technology | Where it runs |
|---|---|---|
| Answer generation | `QCRI/Fanar-1-9B-Instruct`, 4-bit NF4 (bnb, bfloat16 compute) | local GPU (`DEVICE`, default `cuda:0`) |
| Token-level UQ | entropy / collision entropy / LogTokU / GLU + late-layer signals | same GPU as the model |
| Auth | JWT (HS256, `python-jose`) + bcrypt password hashing | this API, `users` table |
| Chat persistence | PostgreSQL `public` schema: `chats`, `messages`, `message_versions` | local Postgres (`DATABASE_URL`) |
| Uploaded chat images | Azure Blob Storage | external (`AZURE_STORAGE_CONNECTION_STRING`) |
| Text embeddings | `BAAI/bge-m3` (multilingual — handles Arabic queries) | local GPU (`RAG_DEVICE`, default `cuda:0`) |
| Sign-image embeddings | `google/siglip-so400m-patch14-384` | local GPU (`RAG_DEVICE`) |
| Vector store | Postgres + pgvector, schema `rag` (tables `rag.chunks`, `rag.signs`) | local Postgres (`DATABASE_URL`) |
| Vision LLM | GPT-4.1-mini (Azure OpenAI) | external API, low volume |
| Sign image files | PNG crops under `data/rag/extracted/<part>/figures/` | local disk, served by this API |

The `rag` schema is fully isolated from the app's `public` schema
(`users`, `chats`, `messages`, `message_versions`) — ingestion never reads
or writes those.

### Request flows

**A. Text question (`use_rag=true`)**
1. Query embedded with BGE-M3 → top chunks fetched from `rag.chunks`
   (cosine search in pgvector).
2. Chunks formatted as sources (with book page numbers) into a grounded
   prompt; Arabic questions force an Arabic answer.
3. Fanar generates the answer; UQ computed per token.

**B. Photo of a sign attached**
1. The image is embedded with SigLIP and matched against every sign crop in
   `rag.signs`.
2. If the best cosine score ≥ `RAG_SIGN_MATCH_THRESHOLD` (default 0.5), the
   top `RAG_SIGN_CANDIDATES` (default 5) candidates go to GPT-4.1-mini, which
   sees the user photo + the candidate crops and picks the true match (or
   rejects all). This catches near-twins like *No Parking* vs *No Stopping or
   Parking*.
3. On a match: the sign's **exact printed name**, category and page are
   injected as a retrieval source, and the catalog crop is attached to the
   response (`data.images[]`).
4. If the image is not a known sign, the system falls back to a generic
   GPT-4.1-mini description, which is appended to the RAG query (old
   behaviour).

**C. "Show me the X sign" (text only)**
1. A lightweight intent check (keywords like *show / display / image /
   picture / صورة / أرني*) triggers a name search over the sign catalog
   (BGE-M3 over `"name — category"`).
2. Confident matches (score ≥ 0.55; runner-up only if nearly tied) are
   attached to the response as `data.images[]`, and the model is told the
   image is already shown so its wording matches.

**Measured accuracy:** on 12 randomly chosen catalog signs perturbed to look
like phone photos (3× rescale, rotation, blur, JPEG artifacts), SigLIP top-1
matching was 12/12 with scores 0.85–0.97.

## 2. What the system can and cannot do

**Can:**
- Answer driving-theory questions grounded in the guide, with page citations,
  in English and Arabic (language is auto-detected from the question).
- Identify a traffic-sign photo and reply with the sign's name **verbatim as
  printed in the book** (including the book's own typos, e.g.
  `"Pedestrain Prohobited"` — this is intentional: the catalog mirrors the
  source material exactly).
- Return the actual sign image from the book, both proactively (photo match)
  and on request ("show me ...").
- Report per-token and aggregate uncertainty for every generated answer
  (including `total_glu` and `total_logtoku`).
- Hold multi-turn context (the frontend sends prior messages explicitly) and
  **persist chats server-side per user** — list, load, rename and delete
  chats; regenerate any answer (stored as a new version).
- **Stream responses** token-by-token over SSE, with full UQ metrics in the
  final `done` event.
- Authenticate users (signup/login → JWT bearer token); every chat and
  generation endpoint is user-isolated.

**Cannot / current limits:**
- **Only Part 3 of the guide (pages 61–102, the traffic-signs chapter) is
  ingested so far** — 94 text chunks, 185 cataloged signs. Other parts need
  the ingestion commands below (one-time, per part).
- One request at a time per GPU: generation is serialized by a lock; under
  concurrent load, requests queue (latency grows, nothing breaks).
- Non-streaming generation takes roughly 5–30 s depending on answer length
  (UQ requires full hidden-state output). Frontends should prefer the
  streaming endpoint, or show a progress state and use generous HTTP
  timeouts (≥ 120 s). Streaming runs generation twice (a fast streaming pass,
  then a UQ pass), so the `done` event arrives well after the last token.
- `use_rag=false` skips retrieval entirely (plain chat with the model).
- Sign-image attachment (`images[]`) is only on the **non-streaming**
  endpoints for now; streaming responses don't match catalog signs yet.

## 3. Setup

### Requirements
- Python 3.10+, CUDA GPUs (model in 4-bit NF4, roughly half the 8-bit
  footprint; embeddings ~5 GB).
- Postgres with the `pgvector` extension available (used for both the app
  tables and the RAG index).
- Azure OpenAI deployment of GPT-4.1-mini (vision) — only needed for sign
  photo re-ranking, image description and catalog ingestion.
- Azure Blob Storage account — only needed for uploaded chat images.

```bash
pip install -r requirements.txt
cp .env.example .env        # then fill in values
python main.py              # serves on HOST:PORT (default 0.0.0.0:8001)
```

### Key environment variables (`.env`)

| Variable | Meaning | Default |
|---|---|---|
| `DEVICE` | GPU for Fanar | `cuda:0` |
| `RAG_DEVICE` | GPU for BGE-M3 + SigLIP | `cuda:0` |
| `DATABASE_URL` | Postgres URL — app tables (`public`) + RAG vectors (`rag`) | — (required) |
| `JWT_SECRET_KEY` | signs auth tokens — set a long random string in production | dev placeholder |
| `JWT_EXPIRE_MINUTES` | access-token lifetime | `10080` (7 days) |
| `AZURE_STORAGE_CONNECTION_STRING` | blob storage for uploaded chat images | — (required) |
| `AZURE_OPENAI_ENDPOINT` / `AZURE_OPENAI_API_KEY` | vision LLM | — |
| `AZURE_OPENAI_VISION_DEPLOYMENT_NAME` | vision deployment | `gpt-4.1-mini-llm-reliability` |
| `RAG_TOP_RESULTS` | retrieved chunks per query | 2 |
| `RAG_SIGN_CANDIDATES` | candidates sent to the vision re-ranker | 5 |
| `RAG_SIGN_MATCH_THRESHOLD` | min SigLIP cosine to treat an image as a sign | 0.5 |
| `CORS_ORIGINS` | comma-separated allowed origins | localhost:3000/3001/8501 |

### Ingestion pipeline (per book part)

```bash
python -m rag.ingestion.split_pdf                          # once: split book into per-part PDFs
python -m rag.ingestion.extract_pdf --part part_4          # Docling: text + figure crops + page renders
python -m rag.ingestion.segment_figures --part part_4     # split multi-sign collages into single signs
python -m rag.ingestion.build_sign_catalog --part part_4  # GPT-4.1-mini: crop -> printed name (verbatim)
python -m rag.ingestion.build_index                        # embed everything, reload pgvector index
```

Notes:
- `build_sign_catalog --page N` labels a single book page (cheap spot-check
  before spending API calls on a whole part).
- `build_index` always rebuilds the **whole** index from every part found
  under `data/rag/` — safe to re-run anytime.
- Page numbers everywhere refer to the original book numbering.
- Outputs live under `data/rag/` (git-ignored): `splits/`, `extracted/<part>/`
  (page renders, figure crops, Docling document), `sign_catalog/<part>.json`.
- After re-ingestion, restart the API (the store caches counts at startup).

## 4. API reference (what the frontend needs)

Interactive docs: `GET /docs` (Swagger) and `GET /redoc`.

| Endpoint | Auth | Purpose |
|---|---|---|
| `GET /health`, `GET /model_info` | — | service status |
| `POST /auth/signup`, `POST /auth/login` | — | get a JWT |
| `GET /auth/me` | JWT | current user |
| `GET /history` | JWT | all chats of the user, grouped |
| `GET /chats/{chat_id}` | JWT | one chat with messages + versions |
| `PATCH /chats/{chat_id}/rename`, `DELETE /chats/{chat_id}` | JWT | manage a chat |
| `POST /chats/{chat_id}/messages` | JWT | generate (non-streaming), persist |
| `POST /chats/{chat_id}/identify-sign` | JWT | upload an image → matched sign name + crop, no LLM, persist |
| `POST /chats/{chat_id}/find-sign` | JWT | describe a sign → its catalog crop, no LLM, persist |
| `POST /generate_response` | JWT | same as `/messages`, chat_id in the body (legacy shape) |
| `POST /chats/{chat_id}/stream` | JWT | generate, streamed over SSE, persist |
| `POST /chats/{chat_id}/messages/{message_id}/regenerate/stream` | JWT | re-answer as a new version |
| `GET /images/{path}` | JWT | uploaded chat image (owner only) |
| `GET /signs/{sign_id}/image` | — | catalog sign crop (public book content) |

Most responses share one envelope (`message`/`timestamp` are omitted on the
auth and chat endpoints):

```json
{ "success": true, "message": "...", "data": { ... }, "timestamp": "ISO-8601" }
```

Errors (any endpoint) use HTTP status codes plus:

```json
{ "success": false, "code": "bad_request | not_found | ...", "message": "..." }
```

### Authentication

```
POST /auth/signup   { "email", "username", "password" }   → 201
POST /auth/login    { "email", "password" }               → 200
```

Both return `data: { "access_token", "token_type": "bearer", "user": {...} }`.
Send the token on every protected call:

```
Authorization: Bearer <access_token>
```

Missing/expired tokens get `401` (`WWW-Authenticate: Bearer`). Signup
validation: valid email, username 3–30 chars, password ≥ 8 chars; duplicate
email/username → `409`. The backend bcrypt-hashes whatever password string it
receives (the frontend may pre-hash with SHA-256). `GET /auth/me` returns the
token's user (`user_id`, `username`, `email`).

### `GET /health`
`data: { "status": "healthy", "service": "...", "model_loaded": true }` —
poll this before enabling the chat UI; the model takes ~1–2 min to load after
a restart.

### `GET /model_info`
`data: { "model_name", "device", "quantization", "vocab_size" }`.

### `POST /chats/{chat_id}/messages` and `POST /generate_response`

Non-streaming generation, **JWT required**. Both behave identically and
persist the exchange to the user's history; the first takes `chat_id` from
the path, the legacy second from the body/form. They accept **two content
types**:

**(a) `application/json`** — text-only requests:

```json
{
  "chat_id": "abc-123",
  "question": "What does the stop sign mean?",
  "use_rag": true,
  "context": [
    { "role": "user", "content": "previous user message" },
    { "role": "assistant", "content": "previous answer" }
  ]
}
```

**(b) `multipart/form-data`** — when attaching an image. Fields:

| Field | Type | Notes |
|---|---|---|
| `chat_id` | text | required |
| `question` | text | may be empty when the image speaks for itself |
| `use_rag` | text | `"true"` / `"false"` (string, not bool) |
| `context` | text | JSON-encoded array, same shape as above |
| `image` | file | JPEG/PNG of the sign photo |

```js
const form = new FormData();
form.append("chat_id", chatId);
form.append("question", "What is this sign?");
form.append("use_rag", "true");
form.append("context", JSON.stringify(history));
form.append("image", fileBlob);          // field name MUST be "image"
await fetch(`${API}/generate_response`, {
  method: "POST",
  headers: { Authorization: `Bearer ${token}` },
  body: form,
});
```

**Response `data`:**

```json
{
  "chat_id": "abc-123",
  "role": "assistant",
  "generation_type": "text",
  "message": "The stop sign is an equilateral octagon ...",
  "token_data": [
    {
      "token": "The", "au": 0.1, "eu": 0.2, "logtoku": -0.02,
      "reliability": -0.1, "entropy": 0.4, "collision_entropy": 0.3,
      "late_mean_top1_prob": 0.9, "late_agreement_rate": 1.0,
      "late_mean_entropy": 0.2, "reliability_with_hidden_layers": 0.5
    }
  ],
  "total_reliability": -0.42,
  "total_entropy": 1.31,
  "total_collision_entropy": 1.02,
  "total_reliability_with_hidden_layers": 0.18,
  "total_glu": 0.37,
  "total_logtoku": -0.05,
  "images": [],
  "image_url": null,
  "generation_time_seconds": 12.3
}
```

Field guide for the frontend:

- **`generation_type`** — `"text"` (no images) or `"multimodal"`
  (`images[]` is non-empty → render them with the message).
- **`message`** — the answer text. May be Arabic if the question was Arabic
  (render with `dir="auto"`).
- **`images[]`** — traffic-sign attachments:

  ```json
  {
    "sign_id": "part_3_p75_f0_s6",
    "name": "No Entry",
    "category": "B. Phohibition Signs",
    "page": 75,
    "url": "/signs/part_3_p75_f0_s6/image"
  }
  ```

  - `url` is **relative to the API base** — prefix it with the backend origin.
  - `name` is the sign's name exactly as printed in the guide; `page` is the
    book page (useful as a citation chip).
  - Images can appear both when the user *uploaded* a photo (the matched sign)
    and when they *asked to see* a sign.
- **`token_data[]`** — one entry per generated token, in order; tokens
  concatenate to `message`. Use `reliability` / `entropy` per token for
  confidence highlighting (higher entropy = less certain token).
- **`total_*`** — aggregate uncertainty over the worst-10 tokens of the
  answer (`total_glu` comes from a separate hidden-state forward pass).
  Lower `total_reliability` / higher `total_entropy` = shakier answer; a
  frontend can badge answers (e.g. "verify this") based on thresholds it
  calibrates.
- **`image_url`** — set when the user uploaded a photo: the stored copy of
  *their* image (`/images/...`, JWT required to fetch). Distinct from
  `images[]`, which are catalog sign crops (public).
- **`generation_time_seconds`** — wall-clock generation time, also persisted
  with the message.

### Dedicated sign modes: `POST /chats/{chat_id}/identify-sign` and `POST /chats/{chat_id}/find-sign`

Two **non-streaming, generation-free** endpoints for the sign-only chat modes.
Neither runs the LLM — they only do retrieval — so they return in well under a
second and the UQ metrics are all `0`. Both **persist to chat history exactly
like the main modes** (a message row is written, visible in `/history` and
`GET /chats/{chat_id}`), and both return the **same `data` envelope as
`/chats/{chat_id}/messages`** (`GenerateResponse`). The payload lives in
`data.images[]` (the matched catalog crop(s), same shape as everywhere else);
`token_data` is `[]` and every `total_*` is `0` — don't render confidence UI
for these messages.

**`POST /chats/{chat_id}/identify-sign`** — "I have a photo, what sign is it?"

- Content type: **`multipart/form-data`**, field **`image`** (file) **required**.
  `question` is optional (a caption to store alongside the upload).
  `use_rag` / `context` are ignored.
- Runs SigLIP match + GPT-4.1-mini re-rank (same logic as a photo attached to
  the main mode), but returns *only* the identification — no generated answer.
- On a match: `data.message` is a short caption (e.g.
  `This is the "No Entry" sign (B. Phohibition Signs), from page 75 ...`),
  `data.images[]` has the one matched sign, `data.image_url` points to the
  stored copy of the uploaded photo, `generation_type` is `"multimodal"`.
- No match: `data.message` says so, `data.images[]` is `[]`,
  `generation_type` is `"text"`, top-level `message` is `"No matching sign found"`.

```js
const form = new FormData();
form.append("image", fileBlob);          // required; field name MUST be "image"
form.append("question", "");             // optional caption
await fetch(`${API}/chats/${chatId}/identify-sign`, {
  method: "POST",
  headers: { Authorization: `Bearer ${token}` },
  body: form,
});
```

**`POST /chats/{chat_id}/find-sign`** — "Show me the sign for X."

- Content type: **`application/json`** (or multipart); field **`question`**
  (the description) **required** — a `400 bad_request` is returned if empty.
  `use_rag` / `context` / `image` are ignored.
- Runs the catalog name search (BGE-M3 over `"name — category"`); returns up
  to 2 confident matches (score ≥ 0.55, runner-up only if nearly tied).
- Match: `data.message` is a short caption naming the sign(s),
  `data.images[]` carries the crop(s), `image_url` is `null`,
  `generation_type` is `"multimodal"`.
- No match: empty `data.images[]`, `generation_type` `"text"`, top-level
  `message` is `"No matching sign found"`.

```js
await fetch(`${API}/chats/${chatId}/find-sign`, {
  method: "POST",
  headers: {
    Authorization: `Bearer ${token}`,
    "Content-Type": "application/json",
  },
  body: JSON.stringify({ question: "no entry sign" }),
});
```

Note: unlike the keyword-triggered "show me ..." behaviour inside the generic
chat mode (request flow **C** above), `find-sign` does **not** require trigger
words — the whole request is treated as a sign-name query.

### `GET /signs/{sign_id}/image`

Returns the sign's PNG crop from the book (`image/png`). `404` with the
standard error body for unknown ids. The images are small (~10–20 KB), safe
to cache (`sign_id` is stable across requests, but may change after a
re-ingestion of the catalog — don't persist ids long-term, use the `url`
from the response you just received).

### Streaming: `POST /chats/{chat_id}/stream`

Same request shapes as the non-streaming endpoints (JSON or multipart with
`image`), but responds with **Server-Sent Events** (`text/event-stream`).
Each line is `data: {json}\n\n`:

```json
{ "type": "token", "content": "The" }
{ "type": "token", "content": " stop" }
...
{ "type": "done",
  "message_id": "…", "message": "The stop sign …", "chat_id": "…",
  "token_data": [ … ],
  "total_reliability": -0.42, "total_entropy": 1.31,
  "total_collision_entropy": 1.02,
  "total_reliability_with_hidden_layers": 0.18,
  "total_glu": 0.37, "total_logtoku": -0.05,
  "generation_time_seconds": 14.0, "image_url": null,
  "role": "assistant", "generation_type": "text" }
{ "type": "error", "content": "…" }   // only on failure
```

Render `token` events as they arrive; when `done` arrives, replace the
accumulated text with `message` and attach the metrics. The `done` event
comes noticeably after the last token (a second pass computes the UQ
metrics). The exchange is persisted before `done` is emitted, so a reload
right after streaming will find it in `/history`.

### Regeneration: `POST /chats/{chat_id}/messages/{message_id}/regenerate/stream`

JSON body `{ "question", "use_rag", "context" }`. Streams exactly like
above; the result is stored as a **new version** of the message (the
original is backfilled as version 1) and the `done` event carries
`version_num`. All versions come back in `GET /chats/{chat_id}` under each
message's `versions[]`.

### Chat history

- `GET /history` — `data: [ { chat_id, title, last_updated, messages: [...] } ]`,
  newest chat first. Each message carries the same fields as a generation
  response (plus `message_id`, `timestamp`, `image_url`), including any
  `images[]` (matched sign crops) — so signs from all modes, including
  `identify-sign` / `find-sign`, re-render correctly on reload.
- `GET /chats/{chat_id}` — one chat in the same shape, with each message's
  `versions[]` populated.
- `PATCH /chats/{chat_id}/rename` — body `{ "title": "..." }` (≤ 80 chars).
- `DELETE /chats/{chat_id}` — removes the chat, its messages, versions and
  uploaded images. Irreversible.
- Chats are created implicitly the first time a `chat_id` is used; the
  default title is the first question (trimmed to 40 chars).

### Practical frontend notes

- **Auth first:** store the JWT from signup/login and send it on every call
  except `/auth/*`, `/health`, `/model_info` and `/signs/.../image`. A `401`
  means re-login (tokens expire after `JWT_EXPIRE_MINUTES`).
- **Timeouts:** prefer the streaming endpoint; for non-streaming allow
  ≥ 120 s per request and show a typing/loading state. Requests are
  processed one at a time per model — concurrent users queue.
- **CORS:** the frontend origin must be listed in `CORS_ORIGINS`.
- **Context vs. history:** generation uses only the `context` array you send
  (user/assistant turns, oldest first) — keep sending it. Persistence is
  separate: every exchange is stored server-side and can be reloaded via
  `/history` after a refresh or on another device.
- **`use_rag`** should normally be `true`; `false` is plain chat without the
  guide (no citations, no sign features).
- **Greetings** short-circuit retrieval (fast response, no sources).
