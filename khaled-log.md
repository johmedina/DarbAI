## Heading/Subheading Interpretation

For Qatar's document, there were multiple but specific formats for headings and subheadings (shown above) which made it slightly easier to interpret them.

For UAE's document, both headings and subheadings have the same text format, but different colors. However, this was not picked up by docling's pdf extraction, so a different approach was needed.

Both headings and subheadings were labelled as level 1 (headings) by docling but some needed to be given level 2 or 3 (subheadings). The pdf's contents page contained a list of confirmed level 1 headings. These were marked in docling's json file. The rest of the wrongly-interpreted level 1s were marked as null. Then they were all sent to an LLM to reassign them to 2s and 3s semantically.

gpt-4.1-mini struggled with this, so we made the decision to switch to gpt-5.4-mini from then on for all LLM/VLM related tasks.

## "Identify Sign" Feature

The "Identify Sign" feature allows users to upload an image of a traffic sign and receive details on it from the source documents.

The pipeline goes as follows:

1. User uploads an image
2. User's image is converted to an embedding
3. The embedding is used to extract the top 10 candidates for a match from the psql table rag.signs
4. Decision stage: The candidate details and user's image are sent to gpt-5.4-mini to decide which candidate matches the user's image (if any)
5. If a candidate is found to match the user's image, the sign details including the context is sent to Fanar (locally) with a dedicated system to generate a response to the user using only the context for the sign (ideally)
6. If no candidate is found, a hard-coded response is sent to the user telling them this

Stage 5 of the pipeline was not added until later. Previously, the user would send a traffic sign image and the pipeline's interpretation was sent back to them with a meaningless hard-coded message.

A test was done on the pipeline with a detailed report generated in the backend repo. It is important to mention that that test was done on gpt-4.1-mini, so with gpt-5.4-mini, the accuracy scores will likely be higher.

## "Describe the Sign" Feature

The "Describe the Sign" allows the user to enter a sign's visual description or exact name and receive details on the sign(s) from the source documents as well as the sign image.

The pipeline goes as follows:

1. User enters a sign description/name
2. The user's input is converted to an embedding
3. The embedding is used to find the top 10 candidates for a match from the psql table rag.signs by searching both name_embeddings and description_embeddings
4. Decision stage: The user's input and the names and descriptions of all candidates are sent to Fanar (locally) to decide which candidate(s) match the user's input (if any)
5. If candidates are found, their details are sent to Fanar (locally, different call) with a dedicated system prompt to generate a response to the user using only the context for the sign (ideally)
6. If no candidate is found, a hard-coded response is sent to the user telling them this

Previously, stage 2 searched either the name_embeddings column or the description_embeddings column for a match. Which column was searched depended on a dropdown on the frontend where the user selected "search by name" or "search by description". To avoid confusion and improve user experience, we switched to searching both columns and considering the top 10. This change made no noticeable difference in the feature's performance.

Previously, the decision stage was handled by gpt-5.4-mini which was given the images in addition to the description. The switch to Fanar resulted in minor performance degradation since Fanar is only given the names and descriptions.

Similar to "Identify Sign", stage 5 of the pipeline was not added until later. Previously, the user would receive the pipeline's interpretation of the name/description and a meaningless hard-coded message.

Unlike "Identify Sign", however, this feature allowed multiple sign matches. This may occur when the user's description is too vague or fits multiple signs. In such cases, all matching signs are returned and a response is generated describing all of them.

## Issues with "Identify Sign" and "Describe the Sign"

The source documents don't contain a lot of context for many of the traffic signs. This resulted in Fanar hallucinating some of the details it said about the signs. Making the prompt strict also didn't help as Fanar would output two responses in one: one that followed the strict prompt, and another that didn't. Changing the prompt fixed some instances of this and broke others. It is clear that another approach is needed: either to add more context to each sign (from official sources), or store hard-coded responses for each sign. Otherwise, sacrifices to consistency must be made.

## Arabic UI

Added arabic UI by replacing all the hard-coded strings in the frontend with references to an array in `translations.ts` where they are stored instead. `translations.ts` also includes arabic translations of all the text. When the user switches language, the text is retrieved from the correct part of the array. This approach allows additional countries to be added later if needed.
